## Đọc code trước — Logic của generateSlug

```ts
export function generateSlug(name: string): string {
  return name
    .toLowerCase()                      // 1. Viết thường
    .trim()                              // 2. Xóa khoảng trắng đầu/cuối
    .replace(/[^a-z0-9\s-]/g, "")         // 3. Xóa ký tự đặc biệt (giữ a-z, 0-9, space, -)
    .replace(/[\s]+/g, "-")              // 4. Thay space bằng -
    .replace(/-+/g, "-")                  // 5. Collapse nhiều - thành 1
    .replace(/^-|-$/g, "");               // 6. Xóa - ở đầu và cuối
}
```

## 4 test case ĐÃ THÊM

### Test 1: Tên đã là slug hợp lệ

```ts
it("returns a name that is already a valid slug unchanged", () => {
  expect(generateSlug("my-cool-app")).toBe("my-cool-app");
});
```

**Tư duy:** `my-cool-app` đã đúng format slug → không cần thay đổi gì.

### Test 2: Tên có số (số được giữ nguyên)

```ts
it("preserves numbers in the slug", () => {
  expect(generateSlug("App123")).toBe("app123");
});
```

**Tư duy:** Regex `/[^a-z0-9\s-]/g` giữ số 0-9 → `App123` → `app123`.

### Test 3: Chuỗi rỗng

```ts
it("returns empty string for empty input", () => {
  expect(generateSlug("")).toBe("");
});
```

**Tư duy:** `""` → `toLowerCase()` → `""` → `trim()` → `""` → `replace(/[^a-z0-9\s-]/g, "")` → `""` → cuối cùng `replace(/^-|-$/g, "")` → `""`.

### Test 4: Tên có ký tự đặc biệt ở đầu/cuối

```ts
it("strips leading/trailing hyphens after special character removal", () => {
  expect(generateSlug("!@#hello$%^")).toBe("hello");
});
```

**Tư duy:**
```
"!@#hello$%^"
  → lowercase: "!@#hello$%^"
  → trim: "!@#hello$%^"
  → xóa ký tự đặc biệt: "hello"      (các ký tự ở 2 đầu bị xóa)
  → thay space bằng -: "hello"
  → collapse -: "hello"
  → xóa - đầu/cuối: "hello"          ✅
```

## 🎓 Những điều đã học được từ E3

### 1. Test-Driven Development (TDD) là gì?
- Viết test **TRƯỚC** rồi code **SAU** (không làm trong bài này nhưng là khái niệm)
- Hoặc viết test **SAU** khi code đã có (đây là cách trong bài E3)
- Test giúp **xác nhận code hoạt động đúng**

### 2. Đọc code trước khi viết test
- **QUAN TRỌNG:** Đừng viết test "mò" — phải đọc implementation trước
- Hiểu logic → mới biết expect gì
- Hint trong comment: "read `src/lib/utils.ts` to understand"

### 3. Cách đọc code để viết test
- Đọc từ TRÊN xuống DƯỚI, từng bước transform
- Mỗi bước làm gì? → ghi ra giấy
- Input → bước 1 → bước 2 → ... → output

### 4. Edge cases là gì?
- **Edge case** = trường hợp đặc biệt, không phải "bình thường"
- Ví dụ: chuỗi rỗng `""`, ký tự đặc biệt, số, v.v.
- Test tốt = test cả edge cases

### 5. Regular Expression (Regex) trong JS
- `/[^a-z0-9\s-]/g` = giữ lại a-z, 0-9, space, dấu `-`
- `replace(/-+/g, "-")` = thay nhiều `-` thành 1 `-`
- `replace(/^-|-$/g, "")` = xóa `-` ở đầu hoặc cuối

### 6. Vitest syntax
```ts
describe("tên nhóm test", () => {
  it("mô tả test này làm gì", () => {
    expect(hàm_cần_test(param)).toBe(kết_quả_đúng);
  });
});
```

### 7. Bài học lớn nhất
> **Viết test không phải "thử đoán" — mà là "hiểu code rồi xác nhận nó hoạt động đúng như mong đợi".**

### 8. Quan trọng: Test phải PASS
- Viết xong test → chạy `pnpm test` → phải pass hết
- Test fail = code có bug HOẶC expect sai
- CI sẽ fail nếu có test fail
